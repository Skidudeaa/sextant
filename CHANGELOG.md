# Changelog

All notable changes to sextant are recorded here. Entries are ordered newest first.

## 2026-05-06 — TS ESM Node16 resolver fix + watch-stop --root hygiene

Two surgical fixes that close real bugs caught while dogfooding sextant on a downstream TypeScript project. **Resolver (`1b8b723`):** TypeScript projects with `"moduleResolution": "node16" | "nodenext"` must write `import "./foo.js"` in source even when the file on disk is `./foo.ts` — the `.js` refers to the compiled output. Sextant's resolver was treating those specifiers as unresolved and reporting 13% (2/16) on the real repo (originally surfaced as "0/13" in the bug report); the 90% health gate then disabled graph reranking entirely, so the user was running in degraded text-only mode without knowing why. `lib/resolver.js:resolveFileAbs` now accepts an optional `{ importerExt }` and rewrites JS-family extensions to TS-family equivalents (`.js → .ts/.tsx`, `.jsx → .tsx/.ts`, `.mjs → .mts/.ts`, `.cjs → .cts/.ts`) — but only when the importer is itself TS-family, so pure-JS projects keep literal-extension semantics. Literal-first ordering preserves mixed-repo correctness when both `foo.js` and `foo.ts` exist. Threaded through `tryTsconfigPaths` and `tryWorkspacePackage` too, so paths-mapped and workspace imports benefit identically. New fixture `fixtures/ts-esm-node16/` exercises the win case end-to-end: 100% resolution (was 0%), MRR 1.0, nDCG 1.0, **+0.144 graph lift on def-site queries** (graph-on puts canonical `schemas.ts` at rank 1; graph-off buries it at rank 2 behind the consumer `index.ts`). Verified externally on `/root/manus-api-mcp`: pre-fix 13% (2/16) → post-fix 100% (16/16); `schemas.ts` fan-in went 0 → 3. **Watch CLI (`4a89721`):** `commands/watch-start.js` and `commands/watch-stop.js` hardcoded `process.cwd()` and silently ignored `--root` / `--roots` / `--roots-file` — running `watch-stop --root /repoB` from inside repoA stopped repoA's watcher and left repoB's lockfile alone. Both commands now route through the same `rootsFromArgs(process.argv)[0]` parser every other command uses; regression test in `test/watch.test.js` spawns the CLI with `--root` pointing at a tmp dir and `cwd` at a different one, asserts only the target's lockfile + heartbeat are cleaned. Self-eval, Vapor benchmark, and all integration scripts unchanged.

## 2026-05-01 — Swift v1 ship: declarations, relations, scoring, eval corpora

Sextant now indexes Swift source the same way it indexes JS/TS and Python: top-level types, members one nesting level deep, extensions, conformance/inheritance edges. Four ship blockers landed across `83c7d0f` → `a39480c`: span-based declaration identity (`(path, start_byte, end_byte)` PK so the three `update(...)` overloads on `PatientStore` each get their own row), tree-sitter extractor via `web-tree-sitter@0.26.8` against vendored `tree-sitter-swift.wasm` 0.7.1 from `alex-pinkus/tree-sitter-swift` (NOT from the `tree-sitter-wasms` npm — its WASM has an incompatible ABI), parser-failure surfacing in `sextant doctor`/statusline action slot/freshness gate's silent-absence body, two Swift-gated scoring signals (+10% enclosing-type when a query token equals the enclosing type name, +15% extension-target when the query is multi-token / contains `+` / contains the literal word `extension`), and a mixed-language fixture (`fixtures/mixed-eval/`) verifying Swift signals don't unfairly bias against TS/Python neighbors. Polish phase (`083d520`/`02376da`/`c27ac8f`): synthetic 13-case eval corpus at `fixtures/swift-eval/` (MRR 0.958, nDCG 0.977, 13/13), external Vapor 4.121.4 benchmark with 15-query battery + regression-gated comparator (`scripts/eval-swift-external.sh`, `fixtures/vapor-baseline.json`; 15/15 cases, MRR 0.591, neutral graph lift), and `docs/swift-v1-scope.md` documenting "repo-local source orientation only" framing + the 11-item deferral list (compiler-backed semantics, `.swiftinterface`, macros, deep nested types, etc.). Self-eval stayed clean: 19/19 cases, MRR 0.929 vs 0.920 baseline. Two real-corpus findings filed as follow-ups (Skidudeaa/sextant#1 and #2): test-tagged sources outside `Tests/` (`Sources/XCTVapor/`, `Sources/VaporTesting/`) aren't caught by `TEST_PENALTY` and outrank canonical defs on common-name queries; graph lift on Vapor is currently neutral on the three starred pathological-lift queries (`URI`, `init`, `Service`) — both are committed evidence rather than folklore for future scoring tuning.

## 2026-04-30 (evening) — actionable statusline + `sextant doctor` Actions block

We turned the operational state of sextant into a copy-pasteable instruction set, while keeping execution fully manual. The statusline now drops the redundant `(155/155)` fraction and the unexplained `130exp` count, leaving only the at-a-glance signals (◆ resolution %, file count, ⟳ watcher, → injection age, ← last file), and reserves an `⚠ run: sextant <cmd>` slot that appears only when an actionable condition is detected — watcher off/stale or resolution below 90% — surfacing one literal command at a time so each fix uncovers the next. `sextant doctor` gained a top-of-output **Actions** block that exhaustively lists every applicable item (state-dir missing, graph.db missing/empty, watcher dead, heartbeat stale, resolution degraded, settings.json missing) with a `→ sextant <cmd>` line under each. The boundary holds: detection and recommendation live in code, execution stays in your hands.

## 2026-04-30 (later same day) — silent-absence freshness gate at the injection layer

Replaced the elapsed-time "ALERT: INDEX STALE" mechanism with a real-state freshness gate. The previous design checked `hours-since-generated_at` and shipped the full structural body anyway — which (a) cried wolf on idle but unchanged repos, training the LLM to ignore the alert, and (b) still leaked stale numeric fields when the repo *had* changed. The new gate (`lib/freshness.js`, `lib/cli.js applyFreshnessGate`) keys staleness to actual repo state — git HEAD, `git status` fingerprint (filtered to exclude sextant's own `.planning/` writes), scanner version, and graph-schema version — and on stale detection sanitizes the injection by construction: no hotspots, no fan-in numbers, no entry points, no module-type histograms, no health percentages, no value derived from `graph.db`. The minimal stale body retains only fields that derive from the live filesystem and `git` directly (root, branch+HEAD, build-system signals, recent git commits) plus a terse "Structural claims unavailable this turn — rescan {requested|pending|unavailable}" marker that honestly reflects the atomic single-flight rescan state. Async rescan is triggered with `--allow-concurrent --force` and is safe under the mtime-gated cache from earlier today (the watcher's RAM copy gets invalidated on the spawned scan's persist instead of clobbering it). Telemetry (`lib/telemetry.js`) records `freshness.stale_hit` and `freshness.blackout_turn` events to `.planning/intel/telemetry.jsonl` for the future Option-5 adaptive sync/async decision — without this dataset, "telemetry-driven later" becomes hand-waving. 459 unit tests pass (22 new), 5 integration scripts pass, 19/19 eval cases pass. Deliberately deferred: the synchronous-rescan-with-budget path (Option 2) and the per-repo adaptive sync/async chooser (Option 5); both can plug in once telemetry shows when sync is safe.

## 2026-04-30 — mtime-gated `loadDb` cache to fix cross-session staleness

We diagnosed and fixed a cross-session correctness bug in sextant: each Claude Code session's MCP server held a process-global in-memory cache of `graph.db` that was never invalidated, so two concurrent sessions in the same project would silently diverge on `sextant_search` results — one serving its session-start snapshot indefinitely while the watcher updated disk for everyone else. The fix landed as commit `d22fb79`: `loadDb()` now stats `graph.db` on each call and evicts the cache when disk is strictly newer, while `persistDb()` bumps the cached mtime after its atomic rename so writers don't self-invalidate. 446 unit tests + 5 integration scripts + 19/19 eval cases still pass, with eval numbers byte-identical to baseline (the gate is a no-op for retrieval ranking).
