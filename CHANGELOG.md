## 2026-07-01 ŌĆö blast-radius lane: action-time injection after edits (docs/016 Sprint 1)

Sextant's first ACTION-TIME injection: after the agent edits a file, the PostToolUse hook may emit one factual note ŌĆö the file's dependents not yet touched this session plus its top git co-change partners ŌĆö via the `hookSpecificOutput.additionalContext` JSON envelope. This targets the "missed blast radius" orientation-failure class named in the project mission but previously unserved: prompt-time retrieval answers "where is X", not "what breaks if I change X".

Grounded by four recon spikes with pre-registered kill criteria (`docs/016-phase0-recon.md`):
- **R1 field-verified the channel** (Claude Code 2.1.198, marker experiment): plain PostToolUse stdout goes to the debug log ONLY ŌĆö the long-standing CLAUDE.md caveat claiming it "can reach the transcript/context" was WRONG (now corrected, Visibility Model table updated). The JSON envelope IS injected as a `<system-reminder>` immediately after the tool call.
- **R2 pass-with-filters**: raw co-change top-50 is 40-86% CHANGELOG/docs junk; filtering to indexed source files leaves ~0% junk, and 37% of sextant's own filtered pairs have NO import edge (hub-orchestrated siblings ŌĆö signal the import graph structurally cannot see). Both v1 requirements shipped: the `isIndexable`+scan-membership pair filter and hub dampening (partner degree > 25 dropped).
- **R3 killed live self-tuning** (88% single-source blocks = no lever for a per-source multiplier; dead-tie simulation; 19/6 lifetime samples on the thin sources; rolling transcript corpus). Fallback shipped: `sextant tune` ŌĆö REPORTING-ONLY per-source open rates with 95% Wilson intervals and nŌēź30 prior-eligibility; no scoring weight reads it.
- **R4**: no hook channel to Task/workflow subagents exists (0/200 transcripts) ŌĆö subagent orientation excluded from this sprint's scope.

Mechanics: `lib/cochange.js` parses `git log -n3000 --name-only --no-merges` at bulk-scan finalize into `cochange_pairs`/`cochange_degree` (wholesale replace, countŌēź3 floor, 5000-pair cap, NUL pair keys); SCHEMA_VERSION 2ŌåÆ3. Emitter discipline: fan-in Ōēź 3 or partner confidence Ōēź 0.4 to speak at all; once per (session, file); Reads and repeat edits silent; content-stale graph emits NOTHING (silent absence); freshness's git cost paid only on would-emit turns; never throws. Telemetry: `blastradius.injected` + audit section in `sextant telemetry`. Per-session state at `.planning/intel/.blastradius.<sessionKey>`.

**Self-caused-drift exception ŌĆö found by the headless gate, not by dogfooding.** In-session emissions "worked" throughout development (sextant's own watcher kept re-stamping scan state), but the headless end-to-end run in a freshly-wired watcher-less repo came back ABSENT: the agent's own edit makes the tree content-stale at exactly the moment the note should fire, so strict freshness suppression killed the lane for its flagship use case. Fix: `recordScanState` persists the filtered dirty-path list (`scanned_status_files`, Ōēż400 paths) atomically with the status hash; `freshness.isSelfCausedStatusDrift` lets the emitter proceed when status-only drift is confined to session-touched files Ōł¬ the edited file (claims are about OTHER files, which haven't moved). Foreign drift / moved HEAD / missing list / any error ŌåÆ strict suppression, as before. Re-run headless gate: the agent received and quoted the note.

Field note: the lane's first real injection fired during its own development session ŌĆö on the very edit that completed the emitter ŌĆö and correctly surfaced the two dependent test files that needed updating next.

Gates: unit 845/845 + integration scripts green, CLI self-eval byte-identical (21/21, MRR 0.900, nDCG 0.920, lift +0.012 ŌĆö scoring untouched), and a HEADLESS end-to-end verification (fresh repo ŌåÆ `sextant init` ŌåÆ `sextant scan` ŌåÆ agent edits a fan-in-4 file ŌåÆ agent quotes the injected note verbatim) ŌĆö the gate that caught the self-caused-drift bug in-session dogfooding missed. New tests: 15 (cochange lib) + 10 (emitter incl. self/foreign drift) + 6 (tune) + 4 (freshness drift) + 1 (telemetry audit).

## 2026-05-29 ŌĆö cooperative watcher pause (scan/watcher coexistence)

`sextant scan`/`rescan` previously hard-refused while the file watcher was live (two writers race on graph.db from independent sql.js in-memory copies ŌåÆ last-rename-wins clobber). Since the watcher auto-starts, that refusal fired on the common path. Replaced it with a cooperative pause: the scan drops a `.scan_in_progress` marker (`lib/freshness.js`: markScanInProgress/clearScanMarker/isScanInProgress, 90s stale window matching the heartbeat-liveness threshold, refreshed every 10s in `commands/scan.js` onProgress, cleared pid-aware in finally); the watcher advertises `scanPauseProtocol` in its heartbeat (`watch.js` + surfaced by `lib/cli.js:getWatcherStatus`) and DEFERS its graph.db writes while the marker is fresh, resuming after ŌĆö its post-scan persist reloads the scan's result via the mtime-gated `loadDb()` instead of clobbering it. `scan` now refuses only when the running watcher predates the protocol (no `scanPauseProtocol` field ŌĆö an old in-memory process); a current watcher ŌåÆ scan proceeds without `--allow-concurrent`. The heartbeat reflects the RUNNING process's code (module-load constant), so a fresh on-disk watch.js can't fool scan while an old process runs.

An adversarial review (3 independent agents reproducing attacks against the real code) caught a HIGH bug in the first cut: the deferral was only in `watch.js:flush()`, which gates the `intel.updateFile` *call* ŌĆö but the actual write fires from a separate already-armed debounced timer in `intel.js` (scheduleGraphPersist ŌåÆ persistGraphUnlocked ŌåÆ graph.persistDb) with no scan check, so a timer armed in the ~750ms before the marker appeared still fired mid-scan and clobbered (live repro: 0/5 scan files survived). Fixed by moving the pause check onto the timer's persist callback (`intel.js:scheduleGraphPersist`), guarding only the incremental path ŌĆö the bulk-scan's own persistGraphUnlocked never defers. Confirmed: the watcher's armed timer now defers while the marker is fresh and persists its queued change after it clears. Defense in depth: `graph.js:persistDb` moves its `loadDb` INSIDE the write lock, closing a load-then-export TOCTOU for any other concurrent writer (two `--allow-concurrent` scans, future MCP writes). The review rated liveness/crash-recovery SAFE (re-arm never strands `pending`; 90s TTL bounds a crashed-scan freeze; long-scan refresh has a 9├Ś margin) and the two-concurrent-scan marker race LOW (closed anyway via the pid-aware clear). Literal-first ordering preserves mixed-repo correctness when both `foo.js` and `foo.ts` exist. ## 2026-04-30 ŌĆö mtime-gated `loadDb` cache to fix cross-session staleness

We diagnosed and fixed a cross-session correctness bug in sextant: each Claude Code session's MCP server held a process-global in-memory cache of `graph.db` that was never invalidated, so two concurrent sessions in the same project would silently diverge on `sextant_search` results ŌĆö one serving its session-start snapshot indefinitely while the watcher updated disk for everyone else. The fix landed as commit `d22fb79`: `loadDb()` now stats `graph.db` on each call and evicts the cache when disk is strictly newer, while `persistDb()` bumps the cached mtime after its atomic rename so writers don't self-invalidate. 446 unit tests + 5 integration scripts + 19/19 eval cases still pass, with eval numbers byte-identical to baseline (the gate is a no-op for retrieval ranking).
