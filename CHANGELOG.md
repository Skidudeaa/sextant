## 2026-06-09 — exported_symbol precision: term-quality gate + no test-floor (docs/012 fixes 1+2)

The benefit re-measurement left one named precision gap: exported_symbol surfacings earned opens at 3.3% vs text_only's 9.4%. Instance-level diagnosis (`docs/012-exported-symbol-gap.md`) showed the lane is sextant's *strongest* signal when real (code-shaped non-test matches: 22%) — 95% of firings were junk: generic English words (`session`, `plan`, `out`, `pass`) exact-matching pytest fixtures and test constants, then def-floored to rank 1.

**Term-quality gate (fix 1)** in both injection lanes (`lib/graph-retrieve.js` Layer 1 hook path, `lib/retrieve.js` export-graph CLI/MCP path): a non-code-shaped term (`utils.isCodeShapedTerm`) earns export injection only when the target file's fan-in clears `EXPORT_INJECT_MIN_FANIN` (5) or the match is exact-case on a case-distinctive name. **No def-floor for test paths (fix 2)** in `mergeResults`: a test-path def signal keeps its penalized graph score but is never lifted onto the zoekt scale (0/89 opened; the discounted floor ≈525 beat zoekt's ~500 base). Fixes 3 (skip lane on borderline confidence) and 4 (case-strictness) from the diagnosis are subsumed by the gate.

Pre-validated by simulating the gate on the 214 historical instances (kills 175 junk surfacings, keeps 6/7 real hits, predicted open rate 15.4%), then verified live on somaNotes' graph.db (`client`/`session`/`out`/`pass` → no injection; `user` → `database/models.py` fan-in 90 survives; `accept_note` survives). All ship gates byte-identical: self-eval 21/21, python-eval 7/7, Vapor CLI+hook diff PASS, hook-eval means unchanged (the two single-case failures reproduce at baseline — pre-existing). Unit 792/792. In-field confirmation accrues via `eval-trajectory` per-source coverage on post-ship sessions.

## 2026-06-06 — benefit proof: trajectory replay + injection-OFF holdback arm (`6857673` → `392c270`)

For its entire life sextant could prove *no-regression* (MRR/nDCG/graphLiftNDCG on synthetic fixtures) but never *benefit* on real agent behavior — the structural hollow-verification trap the 009 synthesis named as load-bearing. This session closed it with two complementary instruments and a verified proof report (`docs/010-benefit-proof.md`).

**Offline trajectory replay (009 #12) — `lib/trajectory.js` + `sextant eval-trajectory`.** Replays real Claude Code session transcripts (`~/.claude/projects/**/*.jsonl`): parses every injection (`<codebase-retrieval>` / `<codebase-intelligence>` in `attachment` records) and every subsequent file-open (`tool_use` Read/Edit/Write/MultiEdit/NotebookEdit), matches opens to surfaced paths (repo-relative + basename), and headlines a **permutation-null LIFT** — actual coverage vs the coverage of a plausible random same-repo surfaced set — because raw coverage is uninterpretable (the agent opens central files regardless). Result on 74 sessions / 7 repos / 4,216 opens: **query-aware retrieval 1.98× lift (6.8% vs 3.4% null)**; static summary only 1.34× (the recency correlation trap — "recent changes" lists files already being edited); median first-touch rank 2; positive in every multi-session repo; robust to dropping the two largest (1.39×); size-matched null 1.96× (not a cardinality artifact). Excludes nested `subagents/`/`workflows/` transcripts (inherited context, not real orientation). Reports per-source coverage so each surfacing signal is measurable individually. **Adversarially verified:** a 6-agent workflow rebuilt the measurement from scratch and attacked six validity threats; the synthesis lead reproduced and *overturned* both "refuted" verdicts (a "fairer" null was drawing from already-opened files — upward-biased by construction; a static "refutation" was an estimator mismatch that agreed on direction). Honest caveat retained everywhere: correlational, not yet causal; precision-flavored (low coverage = precision headroom, not "broken").

**Injection-OFF holdback arm (009 #1 follow-up) — the causal upgrade.** `commands/hook-refresh.js:decideArm` chooses armed|holdback per turn. A holdback turn still runs retrieval and persists the set it *would* have surfaced (tagged `arm:"holdback"`) + fires `retrieval.holdback`, but withholds the `<codebase-retrieval>` block and falls back to the static summary (orientation preserved — the arm withholds the graph-authority *contribution*, not sextant entirely). Deterministic for tests via `SEXTANT_HOLDBACK_FORCE` / stdin `_holdbackForce`. Enabled at 20% on this repo's own git-tracked `.claude/settings.json` to dogfood the baseline; a local cron (`scripts/check-holdback-benefit.sh`) watches `sextant telemetry --json` and logs a READY block once the holdback arm has ≥30 scored opens AND `benefitDelta` is computed (local because the telemetry is gitignored — a remote routine clones the code but not the data).

## 2026-05-29 — cooperative watcher pause (scan/watcher coexistence)

`sextant scan`/`rescan` previously hard-refused while the file watcher was live (two writers race on graph.db from independent sql.js in-memory copies → last-rename-wins clobber). Since the watcher auto-starts, that refusal fired on the common path. Replaced it with a cooperative pause: the scan drops a `.scan_in_progress` marker (`lib/freshness.js`: markScanInProgress/clearScanMarker/isScanInProgress, 90s stale window matching the heartbeat-liveness threshold, refreshed every 10s in `commands/scan.js` onProgress, cleared pid-aware in finally); the watcher advertises `scanPauseProtocol` in its heartbeat (`watch.js` + surfaced by `lib/cli.js:getWatcherStatus`) and DEFERS its graph.db writes while the marker is fresh, resuming after — its post-scan persist reloads the scan's result via the mtime-gated `loadDb()` instead of clobbering it. `scan` now refuses only when the running watcher predates the protocol (no `scanPauseProtocol` field — an old in-memory process); a current watcher → scan proceeds without `--allow-concurrent`. The heartbeat reflects the RUNNING process's code (module-load constant), so a fresh on-disk watch.js can't fool scan while an old process runs.

An adversarial review (3 independent agents reproducing attacks against the real code) caught a HIGH bug in the first cut: the deferral was only in `watch.js:flush()`, which gates the `intel.updateFile` *call* — but the actual write fires from a separate already-armed debounced timer in `intel.js` (scheduleGraphPersist → persistGraphUnlocked → graph.persistDb) with no scan check, so a timer armed in the ~750ms before the marker appeared still fired mid-scan and clobbered (live repro: 0/5 scan files survived). Fixed by moving the pause check onto the timer's persist callback (`intel.js:scheduleGraphPersist`), guarding only the incremental path — the bulk-scan's own persistGraphUnlocked never defers. Confirmed: the watcher's armed timer now defers while the marker is fresh and persists its queued change after it clears. Defense in depth: `graph.js:persistDb` moves its `loadDb` INSIDE the write lock, closing a load-then-export TOCTOU for any other concurrent writer (two `--allow-concurrent` scans, future MCP writes). The review rated liveness/crash-recovery SAFE (re-arm never strands `pending`; 90s TTL bounds a crashed-scan freeze; long-scan refresh has a 9× margin) and the two-concurrent-scan marker race LOW (closed anyway via the pid-aware clear). Gates: unit 665/665 (+ a clobber-deferral regression test driving the real timer path, + marker/pid-clear/getWatcherStatus unit tests); self-eval MRR 0.908 / 21-21, python-eval CLI 7/7, Vapor CLI+hook delta +0.0000.

## 2026-05-06 — TS ESM Node16 resolver fix + watch-stop --root hygiene

Two surgical fixes that close real bugs caught while dogfooding sextant on a downstream TypeScript project. **Resolver (`1b8b723`):** TypeScript projects with `"moduleResolution": "node16" | "nodenext"` must write `import "./foo.js"` in source even when the file on disk is `./foo.ts` — the `.js` refers to the compiled output. Sextant's resolver was treating those specifiers as unresolved and reporting 13% (2/16) on the real repo (originally surfaced as "0/13" in the bug report); the 90% health gate then disabled graph reranking entirely, so the user was running in degraded text-only mode without knowing why. `lib/resolver.js:resolveFileAbs` now accepts an optional `{ importerExt }` and rewrites JS-family extensions to TS-family equivalents (`.js → .ts/.tsx`, `.jsx → .tsx/.ts`, `.mjs → .mts/.ts`, `.cjs → .cts/.ts`) — but only when the importer is itself TS-family, so pure-JS projects keep literal-extension semantics. Literal-first ordering preserves mixed-repo correctness when both `foo.js` and `foo.ts` exist. Threaded through `tryTsconfigPaths` and `tryWorkspacePackage` too, so paths-mapped and workspace imports benefit identically. New fixture `fixtures/ts-esm-node16/` exercises the win case end-to-end: 100% resolution (was 0%), MRR 1.0, nDCG 1.0, **+0.144 graph lift on def-site queries** (graph-on puts canonical `schemas.ts` at rank 1; graph-off buries it at rank 2 behind the consumer `index.ts`). Verified externally on `/root/manus-api-mcp`: pre-fix 13% (2/16) → post-fix 100% (16/16); `schemas.ts` fan-in went 0 → 3. **Watch CLI (`4a89721`):** `commands/watch-start.js` and `commands/watch-stop.js` hardcoded `process.cwd()` and silently ignored `--root` / `--roots` / `--roots-file` — running `watch-stop --root /repoB` from inside repoA stopped repoA's watcher and left repoB's lockfile alone. Both commands now route through the same `rootsFromArgs(process.argv)[0]` parser every other command uses; regression test in `test/watch.test.js` spawns the CLI with `--root` pointing at a tmp dir and `cwd` at a different one, asserts only the target's lockfile + heartbeat are cleaned. Self-eval, Vapor benchmark, and all integration scripts unchanged.

## 2026-04-30 (later same day) — silent-absence freshness gate at the injection layer

Replaced the elapsed-time "ALERT: INDEX STALE" mechanism with a real-state freshness gate. The previous design checked `hours-since-generated_at` and shipped the full structural body anyway — which (a) cried wolf on idle but unchanged repos, training the LLM to ignore the alert, and (b) still leaked stale numeric fields when the repo *had* changed. The new gate (`lib/freshness.js`, `lib/cli.js applyFreshnessGate`) keys staleness to actual repo state — git HEAD, `git status` fingerprint (filtered to exclude sextant's own `.planning/` writes), scanner version, and graph-schema version — and on stale detection sanitizes the injection by construction: no hotspots, no fan-in numbers, no entry points, no module-type histograms, no health percentages, no value derived from `graph.db`. The minimal stale body retains only fields that derive from the live filesystem and `git` directly (root, branch+HEAD, build-system signals, recent git commits) plus a terse "Structural claims unavailable this turn — rescan {requested|pending|unavailable}" marker that honestly reflects the atomic single-flight rescan state. Async rescan is triggered with `--allow-concurrent --force` and is safe under the mtime-gated cache from earlier today (the watcher's RAM copy gets invalidated on the spawned scan's persist instead of clobbering it). Telemetry (`lib/telemetry.js`) records `freshness.stale_hit` and `freshness.blackout_turn` events to `.planning/intel/telemetry.jsonl` for the future Option-5 adaptive sync/async decision — without this dataset, "telemetry-driven later" becomes hand-waving. 459 unit tests pass (22 new), 5 integration scripts pass, 19/19 eval cases pass. Deliberately deferred: the synchronous-rescan-with-budget path (Option 2) and the per-repo adaptive sync/async chooser (Option 5); both can plug in once telemetry shows when sync is safe.

## 2026-04-30 — mtime-gated `loadDb` cache to fix cross-session staleness

We diagnosed and fixed a cross-session correctness bug in sextant: each Claude Code session's MCP server held a process-global in-memory cache of `graph.db` that was never invalidated, so two concurrent sessions in the same project would silently diverge on `sextant_search` results — one serving its session-start snapshot indefinitely while the watcher updated disk for everyone else. The fix landed as commit `d22fb79`: `loadDb()` now stats `graph.db` on each call and evicts the cache when disk is strictly newer, while `persistDb()` bumps the cached mtime after its atomic rename so writers don't self-invalidate. 446 unit tests + 5 integration scripts + 19/19 eval cases still pass, with eval numbers byte-identical to baseline (the gate is a no-op for retrieval ranking).
